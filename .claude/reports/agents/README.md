# מערכת דיווחים — סוכני TripMaster

> **חובה לכל סוכן**: בתום כל פעולה מהותית (feature, bug fix, review, refactor, research), כתוב דוח כאן לפני סיום ה-turn.

## למה?

1. **שקיפות** — אלי (CEO) יכול לראות מה כל סוכן עושה, בלי שנצטרך לחזור על זה בצ׳אט.
2. **ביקורת עצמית** — אם כל סוכן מתעד מה הוא חושב על התוצאה שלו, בעיות צצות מוקדם.
3. **ניהול מקהלה** — הסוכן הראשי (`tripmaster-manager`) סוקר את כל הדוחות מדי שבוע, מזהה חיכוך בין סוכנים, חוסר ידע, או צורך בגיוס מומחה חדש.
4. **זיכרון לאורך זמן** — הפרויקט יחיה שנים. דוחות טובים היום = הקשר עשיר עוד שנה.

## מבנה תיקיות

```
.claude/reports/agents/
├── README.md                    ← הקובץ הזה
├── TEMPLATE.md                  ← תבנית לכל דוח
├── tripmaster-manager/          ← דוחות של הסוכן הראשי
├── ui-designer/
├── destinations-expert/
├── financial-expert/
├── logistics-expert/
├── halacha-expert/
├── database-expert/
└── meta-reviews/                ← סקירות הסוכן הראשי על עבודת שאר הסוכנים
```

## קונבנציית שמות

`YYYY-MM-DD-<slug-קצר>.md`

דוגמאות:
- `2026-04-17-day-9-command-deck.md`
- `2026-04-18-rls-migration.md`
- `2026-04-20-shabbat-zmanim-research.md`

אם יש כמה דוחות באותו יום מאותו סוכן — הוסף מספר סידורי: `-01`, `-02`.

## מה חייב להיות בדוח?

ראה `TEMPLATE.md`. בקצרה: **6 שדות חובה** — מה ביצעתי, כיצד, תוצאה, הערכה עצמית, חסמים, המלצות להמשך.

## איך הסוכן הראשי משתמש בזה

`tripmaster-manager` מחויב (ראה `.claude/agents/tripmaster-manager.md`) לבצע **Agent Report Review** פעם בשבוע (ביום שישי לפני שבת) ולכתוב דוח מטא ב-`meta-reviews/`. הוא בודק:

1. **איכות ביצוע** — האם כל סוכן מתעד באמת? איזה דוחות שטחיים?
2. **חיכוך בין תחומים** — איפה סוכן ביקש עזרה מחברו? האם החברה הגיבה?
3. **פערים** — נושא שאף סוכן לא נגע בו = צורך לגייס מומחה (למשל: `security-expert`, `marketing-expert`, `content-writer`).
4. **איכות עצמית** — האם הערכות עצמיות כנות? מי תמיד כותב "הצליח מצוין" בלי ביקורת?
5. **מקצב** — סוכן שלא דיווח שבוע = או שאין לו מה לעשות (בעיה), או שלא מזמינים אותו (בעיה אחרת).
