## Executive Summary

אלי הגיש 7 בקשות שמרכיבות קפיצת גרסה של ממש: TripMaster הופך ממערכת תיעוד טיולים לפלטפורמה חכמה שמבינה מה אתה אוכל, לאן אתה נוסע, ואיזה סוג יחסים יש בינך לבין שאר המשתתפים. שלושה צירים עיקריים:

1. **שרשרת המזון** (features 1-4): רשימת קניות דינמית + ספירת סועדים + נרמול רכיבים + הצעות מתכון. המערכת הופכת מ"רשימה שיוצאת מעטה" ל-pipeline סמנטי שמבין שחביתה = ביצים.
2. **העשרת היעד** (feature 5): מנה לאומית, מנהגים, חגים מקומיים, שעות פתיחה בקיעים, סגירות יום ראשון. משנה את `DestinationInfo` schema ואת `destination-generator.ts`.
3. **מודל עסקי דו-ממדי** (feature 6): יצירת טיול מקבלת שני מימדים חדשים — `location_type` (ישראל/חו"ל) ו-`purpose` (לקוח/חברים/משפחה). משנה UI, שפה, ותפריט רכיבים (למשל ב-"משפחה" אין מחיר/רווח).

Feature 7 (ישיבת הצוות) הוא המסמך הזה עצמו + הדוחות האישיים + פרוטוקול הישיבה.

**עקרון מנחה:** לא כותבים שורת קוד אחת לפני שאלי מאשר את המסמך ועונה על 5 החלטות החוסמות בסוף.

## Feature 1 — Shopping List Dynamic Aggregation

### הבעיה
כיום `generateShoppingList` ב-`lib/shopping-generator.ts` אכן מצטבר לפי `ingredient+unit`, אבל ה-UI בנוי סביב "כפתור ייצור" שמכניס שורות חדשות ל-`shopping_items` בכל לחיצה — בלי למחוק ישנות ובלי למזג. התוצאה: כפילויות, סיכום שגוי, וחוסר אמון במערכת.

### הפתרון
- הפיכת `shopping_items` ל-**materialized view מקרוב**: חישוב אוטומטי ב-DB trigger (או בצד הקליינט) בכל שינוי ב-`meal_items`.
- constraint UNIQUE על `(trip_id, ingredient_normalized, unit)` — אי אפשר ליצור כפילות.
- מחיקת הכפתור "צור רשימת קניות" — הוא מתרחש אוטומטית.
- `is_purchased` נשמר על פני מיזוגים (שדה טופל בנפרד ב-upsert).

### DB Changes
```sql
ALTER TABLE shopping_items
  ADD COLUMN ingredient_normalized TEXT,  -- trim + lower + remove niqqud
  ADD COLUMN source_meal_ids UUID[],      -- audit: מאיפה הגיע
  ADD COLUMN auto_generated BOOLEAN DEFAULT true;

CREATE UNIQUE INDEX idx_shopping_unique_per_trip
  ON shopping_items (trip_id, ingredient_normalized, unit)
  WHERE deleted_at IS NULL;
```

### UI Changes
- רשימת הקניות נטענת דרך realtime subscription ל-`meal_items` ו-`shopping_items`.
- ה-badge "נוצר אוטומטית" על שורות שמקורן בארוחות.
- כפתור ידני "הוסף פריט" נשאר — שומר `auto_generated=false` ולא נמחק.

### Owners
- **לוגיסטיקה:** לוגיקת האגרגציה + טריגר
- **DB:** אינדקסים, schema
- **UI:** התנהגות realtime + הודעות

### Depends on
- Feature 3 (ingredient normalization) — **חוסם**. בלי נרמול, ביצים מ"חביתה" וביצים מ"ארוחת בוקר" לא יתמזגו.

## Feature 2 — Meal Diner Count + Smart Title Parsing

### הבעיה
כיום `meals.servings` קיים אך לא בשימוש עקבי. אין איך לרשום "לארוחה הזו באו עוד 4 אורחים". ובנוסף — אין זיהוי סמנטי של כותרת הארוחה.

### הפתרון
- שדה חדש: `meals.diner_count INTEGER` — ברירת מחדל = `totalPeople` של הטיול, ניתן לדריסה ידנית.
- "על האש", "מנגל", "BBQ" בכותרת → פתיחת מגירת הצעות BBQ: כנפיים, שניצלים, פרגיות, סטייקים, ירקות לגריל.
- ירקות (עגבנייה, פלפל, בצל) מתנותבים אוטומטית לקטגוריה `vegetables`.

### DB Changes
```sql
ALTER TABLE meals
  ADD COLUMN diner_count INTEGER,
  ADD COLUMN detected_tags TEXT[];  -- ['bbq','soup','pareve']
```

### UI Changes
- `meal-planner.tsx`: פקד "סועדים" עם `+/-` ליד כל ארוחה. ברירת מחדל = totalPeople.
- `meal-ingredients.tsx`: בזיהוי "על האש" → מציג תבנית מהירה עם checkboxes של 6 פריטי גריל קלאסיים.
- מודל נתונים: טבלת lookup (JSON ב-code בתחילה) `MEAL_TITLE_PATTERNS`:
  ```
  { pattern: /על האש|מנגל|bbq/i, category: 'bbq', suggestions: [...] }
  ```

### Owners
- **לוגיסטיקה:** רשימות ההצעות, חישוב lookup
- **DB:** שדה diner_count
- **UI:** פקד הסועדים, modal ההצעות

### Depends on
- עצמאי ברובו. צריך רק שהאגרגציה ב-Feature 1 תקרא `diner_count` במקום `totalPeople` הגלובלי.

## Feature 3 — Ingredient Normalization (Culinary Knowledge)

### הבעיה
משתמש כותב "חביתה" בכותרת ארוחה, המערכת מכניסה "חביתה" לרשימת הקניות. אבל חביתה זה לא פריט בסופר — צריך ביצים.

### הפתרון
- טבלה חדשה: `ingredient_recipes` — מיפוי מנות לרכיבים + כמויות per-person.
- זרע ראשוני של 30-50 מנות נפוצות: חביתה (2 ביצים), שקשוקה (2 ביצים + 1 עגבנייה + 1/4 פלפל), סלט ישראלי (0.5 עגבנייה + 0.5 מלפפון), מרק עוף, פסטה ברוטב, אורז לבן, וכו'.
- כאשר ארוחה נוספת, המערכת בודקת: האם הכותרת תואמת מנה מוכרת? אם כן — מציעה רכיבים מנורמלים.
- המשתמש יכול לקבל/לדחות/לערוך.

### DB Changes
```sql
CREATE TABLE ingredient_recipes (
  id UUID PK,
  dish_name TEXT UNIQUE,           -- "חביתה"
  aliases TEXT[],                   -- ["omelette","חביתת גבינה"]
  ingredients JSONB,                -- [{name:"ביצים", qty:2, unit:"יח'", category:"parve"}]
  servings_base INTEGER DEFAULT 1,
  source TEXT,                      -- 'seed' | 'llm' | 'user'
  created_at
);
```

### UI Changes
- modal חדש "המערכת הציעה:" כשמזהים התאמה.
- כפתור "ערוך רכיבים" — UX חלק לשינוי כמויות.
- סימון ויזואלי: "רכיבים מחושבים מ-'חביתה'" (badge עם icon ChefHat).

### Owners
- **לוגיסטיקה:** זרע הנתונים, לוגיקת ההתאמה, כמויות per-person
- **DB:** הסכמה, אינדקס על aliases
- **UI:** modal ההצעה

### Depends on
- חוסם את Feature 1 ואת Feature 4.

### Risk
- סיד של 50 מנות בעברית זה עבודה ידנית. שיקול: להשתמש ב-LLM חד-פעמית לייצור הסיד.

## Feature 4 — Recipe Suggestions

### הבעיה
משתמש שם ארוחה "מרק עוף" — מקבל רשימת קניות אך לא יודע *איך* להכין.

### הפתרון
- כפתור "קבל מתכון" על כל meal card (אם כותרת מזוהה).
- בלחיצה: dialog עם מתכון (שלבים + טיפים).
- מקור: (א) טבלה `recipes` פנימית עם מתכונים מוגזמים לסיד, (ב) fallback — קריאת Claude API (sonnet) עם prompt מובנה, (ג) cache לפי `recipe_key` כדי לא לשלם פעמיים.

### DB Changes
```sql
CREATE TABLE recipes (
  id UUID PK,
  dish_name TEXT UNIQUE,
  instructions_md TEXT,
  prep_time_min INTEGER,
  cook_time_min INTEGER,
  difficulty TEXT,                  -- 'easy'|'medium'|'hard'
  tips TEXT[],
  generated_by TEXT,                -- 'seed'|'claude'|'user'
  created_at
);

CREATE INDEX idx_recipes_dish ON recipes (dish_name);
```

### UI Changes
- כפתור "קבל מתכון" עם icon `ChefHat` במסגרת הארוחה.
- Dialog בסגנון `destination-overview` — כרטיסי שלבים עם zebra stripes.
- אם fetched מ-LLM: badge "נוצר על ידי AI" + תאריך.

### Owners
- **לוגיסטיקה:** סיד מתכונים, הגדרת הפרומפט
- **DB:** טבלת recipes + cache
- **UI:** dialog המתכון

### Depends on
- Feature 3 (נרמול ייתן לנו dish_name מקונוני).

### Risk / Decision Point
- **מי משלם על הקריאות ל-Claude API?** 1,000 משתמשים × 2 מתכונים חדשים בחודש × ~3k tokens = ~$30/חודש. Cache תת-שכן.

## Feature 5 — Destination Improvements

### הבעיה
כיום `DestinationInfo` חסר 4 שדות קריטיים: מנה לאומית, מנהגים, חגים מקומיים, שעות פתיחה ליום ראשון וחגים.

### הפתרון — הרחבת הסכמה
```typescript
interface DestinationInfo {
  // ... קיים
  national_dish: { name: string; description: string; where_to_eat?: string[] };
  customs_brief: string;                         // 1-2 משפטים בעברית
  local_holidays: LocalHoliday[];                // ייפלטרו לפי trip.start_date/end_date
  business_hours_notes: {
    sunday: string;                              // "רוב העסקים סגורים" / "פתוח רגיל"
    holidays: string;
    general: string;
    source_date: string;                         // תאריך עדכון
  };
}

interface LocalHoliday {
  name: string;                                  // "Songkran"
  name_he: string;                               // "פסטיבל המים"
  date_range: { start: string; end: string };    // ISO dates
  impact: 'high'|'medium'|'low';                 // מה סגור
  description: string;
}
```

### Logic
- בטעינת עמוד היעד: פילטר `local_holidays` לפי חפיפה עם `trip.start_date..end_date`.
- אם יש חפיפה → באנר בולט בכותרת היעד עם אזהרה.

### UI Changes
- סעיף חדש ב-`destination-overview.tsx`: "תרבות מקומית" עם 3 כרטיסים (מנה, מנהגים, שעות).
- באנר עליון אם יש חג מקומי בטווח.

### Owners
- **יעדים:** איסוף הנתונים, עדכון הפרומפט של generator
- **DB:** הוספת השדות ל-`destinations_cache.data` (JSONB — אין מיגרציה סכמה אמיתית)
- **UI:** סעיף התרבות + באנר

### Risk
- Destinations קיימים (Montenegro/Rome/Athens) יצטרכו תוספת ידנית.
- AI-generated יעדים קיימים — לרענן cache עבורם.

## Feature 6 — Trip Creation: 2-Dimensional Wizard

### הבעיה
כיום `trips` טבלה כוללת `holiday_type` בלבד. אין הבחנה בין נסיעה לעבודה ללקוח לבין טיול משפחתי, ואין הבחנה ישראל/חו"ל (שמייתרת בית חב"ד, המרת מטבע, וכו').

### הפתרון
```sql
ALTER TABLE trips
  ADD COLUMN location_type TEXT CHECK (location_type IN ('israel','abroad')) DEFAULT 'abroad',
  ADD COLUMN purpose TEXT CHECK (purpose IN ('customer','friends','family')) DEFAULT 'friends',
  ADD COLUMN billing_enabled BOOLEAN DEFAULT false,  -- רלוונטי ל-friends (אלי בוחר)
  ADD COLUMN fee_amount NUMERIC(10,2),                -- רלוונטי ל-customer
  ADD COLUMN fee_currency TEXT DEFAULT 'ILS';
```

### מודל עסקי — 3 שכבות
| Purpose | Billing | UI שפה | תכונות מוסתרות |
|---|---|---|---|
| `customer` | חובה | "למשלם", "עלות לשירות" | — |
| `friends` | אופציונלי (per-trip) | "לחבר" / "לחברים" | — |
| `family` | אין | "לבן משפחה" | מחיר, רווח, fee. מציגים רק "מי מביא, מי משלם" |

### ישראל vs חו"ל
| Feature | Israel | Abroad |
|---|---|---|
| Chabad lookup | מושבת | פעיל |
| Currency converter | מושבת (ILS בלבד) | פעיל |
| Emergency phone | 100/101/102 | לפי יעד |
| Halachic notes עירוב/מקווה | מקומי מהארץ | נדרש |

### UI Changes — Wizard חדש ליצירת טיול
- מסך 1: סוג מיקום (Israel/Abroad) — 2 כפתורים גדולים.
- מסך 2: מטרה (Customer/Friends/Family) — 3 כרטיסים עם אייקונים.
- מסך 3: פרטים קיימים (שם, יעד, תאריכים, holiday_type).
- מסך 4 (רק ל-customer או friends-with-billing): סכום, מטבע.

### Owners
- **פיננסי:** מודל ה-billing, חישוב fee, שילוב עם expense calculator
- **DB:** עמודות חדשות
- **UI:** wizard בן 4 שלבים, מילון i18n מעודכן

## Risks & Unknowns

1. **Token cost** — פיצ'ר 4 דורש Claude API. צריך להחליט: cache חזק? limit למשתמש?
2. **Ingredient normalization seed quality** — 50 מנות ידניות זה הרבה. אופציה: LLM generation חד-פעמית + בדיקה אנושית.
3. **Family trips vs Supabase billing logic** — אם Family מסתיר מחיר, האם expense_calculator עדיין רץ? כנראה כן, אבל ה-UI לא מציג.
4. **Existing trips retroactive** — trips קיימים צריכים `location_type` ו-`purpose` — נגדיר ברירת מחדל: abroad + friends.
5. **Destinations cache staleness** — שדות חדשים → כל `destinations_cache` צריך refresh. Fresh trigger?
6. **Dynamic shopping realtime cost** — Supabase realtime על free tier מוגבל. צריך למדוד.
